# AfroSeller QA Fixture

Updated: 2026-05-09

## Purpose

`scripts/afroseller-qa-fixture.js` creates a repeatable AfroSeller Social Commerce OS QA baseline without touching real customer data by default. The default run is a dry-run plan. Live smoke runs require an explicit QA user, a clearly named QA business, a fixture tag, and confirmation.

Use this for `/pro/apps/seller/` validation before deeper Seller feature work.

## Safety Rules

- Default mode is dry-run.
- No live writes run unless `AFROSELLER_QA_CONFIRM=1` and `--live` are both present.
- Live writes require `AFROSELLER_QA_USER_ID`, `AFROSELLER_QA_ACCESS_TOKEN`, and a publishable Supabase key.
- Live writes require `AFROSELLER_QA_BUSINESS_NAME` containing `QA`, `test`, `fixture`, or `sandbox`.
- Live writes require `AFROSELLER_QA_TAG` or `--tag` so cleanup can target the fixture.
- Cleanup refuses to run without an explicit fixture tag.
- The fixture uses only fake business, staff, customer, catalog, order, payment, delivery, export, and audit-event content.
- The browser sync bridge must keep using the signed-in user session. No service-role key belongs in browser code.

## Fixture Scope

The dry-run and live fixture cover these tables:

- `seller_businesses`
- `seller_team_members`
- `seller_products`
- `seller_product_variants`
- `seller_stock_movements`
- `seller_customers`
- `seller_customer_labels`
- `seller_orders`
- `seller_order_items`
- `seller_payments`
- `seller_deliveries`
- `seller_message_templates`
- `seller_exports`
- `seller_audit_events`

Live audit rows are trigger-generated by the database. The fixture does not try to insert `seller_audit_events` directly through the browser-user lane.

## Product Truth Baseline

Local:

- AfroSeller runs from browser storage under `afroseller_social_commerce_os_v1`.
- Local state includes the business profile, catalog preview settings, variants, stock movements, customers, customer labels, orders, manual payment notes, delivery notes, WhatsApp copy text, daily close drafts, saved close history, and CSV/Markdown/HTML exports.
- Local exports still come from the browser copy.

Saved to account:

- Signed-in Pro users can import one Seller workspace through `assets/js/lib/afroseller-sync.js`.
- Account-backed import writes business, product, variant, customer, label, order, order item, payment, delivery, and stock movement rows through normal RLS.
- Cloud metadata is stored separately under `afroseller_social_commerce_os_cloud_v1`.
- The QA fixture can additionally create fake message-template and export rows for a tagged QA business.
- Catalog preview settings, saved close history, finance handoff drafts, local discount/refund review fields, inventory valuation downloads, reorder-list downloads, and the newest fulfillment labels remain browser-first until the account bridge and live constraints are expanded.

Not implemented:

- Automatic two-way sync.
- Conflict resolution.
- Background sync or offline replay.
- Committed generated TypeScript bindings for this app.
- Storage bucket creation or media upload.
- Payment provider collection or verification.
- WhatsApp Business API sending.
- Live storefront publishing.
- Delivery-provider booking or tracking.
- Background jobs for daily close or receipts.

Must not be claimed:

- Real payment provider receipts.
- Live storefront orders.
- Delivery-provider tracking.
- Automatic cloud backup.
- Account save success unless signed-in rows were verified.
- Any browser use of service-role credentials.

## Commands

Dry-run plan:

```powershell
npm run afroseller:qa -- --dry-run
```

Dry-run cleanup plan for a known tag:

```powershell
npm run afroseller:qa -- --cleanup --tag=qa_seller_YYYYMMDDHHMMSS_xxxxxx --dry-run
```

Live fixture smoke, only after a safe QA user is selected:

```powershell
$env:AFROSELLER_QA_CONFIRM = "1"
$env:AFROSELLER_QA_TAG = "qa_seller_20260509_safe_smoke"
$env:AFROSELLER_QA_USER_ID = "<safe QA auth user id>"
$env:AFROSELLER_QA_BUSINESS_NAME = "[QA Fixture] AfroSeller Sandbox"
$env:AFROSELLER_QA_ACCESS_TOKEN = "<signed-in QA user access token>"
$env:SUPABASE_AUTH_ANON_KEY = "<publishable anon key>"
npm run afroseller:qa -- --live
```

Live cleanup for the exact fixture tag:

```powershell
$env:AFROSELLER_QA_CONFIRM = "1"
$env:AFROSELLER_QA_TAG = "qa_seller_20260509_safe_smoke"
$env:AFROSELLER_QA_USER_ID = "<safe QA auth user id>"
$env:AFROSELLER_QA_BUSINESS_NAME = "[QA Fixture] AfroSeller Sandbox"
$env:AFROSELLER_QA_ACCESS_TOKEN = "<signed-in QA user access token>"
$env:SUPABASE_AUTH_ANON_KEY = "<publishable anon key>"
npm run afroseller:qa -- --cleanup --live
```


## Live Inspection Note

For this baseline, live Supabase inspection should happen before repo edits when available. Use the configured Supabase MCP server to confirm the seller tables and RLS shape, then keep any live fixture action separate from repo changes in the final report.
